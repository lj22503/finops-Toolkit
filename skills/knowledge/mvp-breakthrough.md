# MVP 最小可行性破局法

## Overview

在不确定的新业务/新功能上，先用最小成本验证假设，再决定是否加大投入。

## When to Use

- 尝试新的内容方向时
- 测试新的活动形式时
- 探索新的用户群体时

## MVP Process

```
假设 → 最小实验 → 收集数据 → 决策
```

### Step 1: Define Hypothesis

**格式：**
- "我们认为 [目标用户] 会因为 [某种价值] 而进行 [某种行为]"
- "验证标志：[可量化指标]"

**示例：**
- "我们认为新用户会因为'免费体验金'而完成首次开户"
- "验证标志：领券用户中30%完成开户"

### Step 2: Build MVP

**原则：**
- 最小功能集
- 最短路径
- 最低成本

**示例：**
- 不做App → 用小程序先验证
- 不做大活动 → 先发3篇内容测试
- 不做全套 → 先做核心路径

### Step 3: Measure

**指标：**
- 转化率
- 用户反馈
- 完成率

### Step 4: Decide

- 数据支持假设 → 加大投入
- 数据不支持 → 快速放弃，节省资源

## Application

**示例：**
- 新专栏测试 → 先发3篇文章看阅读量和评论反馈
- 新活动形式 → 先小规模试点再扩大
- 新用户群体 → 先定向触达100人看响应率

## Anti-Patterns

- MVP 变成"永远不上线完整版"
- 过度投入在失败的方向上
- 没有明确的决策标准

## Gate Rule

重大投入（预算/人力）前必须验证MVP假设。
